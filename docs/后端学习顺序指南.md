## Phase 1：项目骨架（3 个文件）

> 📌 **目的**：搞清楚项目怎么启动、依赖了哪些库、配置文件里有什么

### 🔹 第 1 个：`pom.xml`

**路径**：`backend/pom.xml`
**作用**：Maven 项目定义文件，声明所有依赖

**阅读要点**：
- `SpringBoot 2.7.18` — 父工程版本
- `mybatis-plus 3.5.3.1` — 数据库 ORM（不需要写 SQL XML）
- `hutool 5.8.22` — 国产 Java 工具库（字符串处理、随机数等）
- `jjwt 0.9.1` — JWT 令牌生成/解析
- `fastjson2` — JSON 解析（阿里出品，用于 LLM API 的 JSON 构造）
- `okhttp` — HTTP 客户端（用于调用 DeepSeek/Anthropic API）
- `lombok` — 省去 getter/setter 样板代码

**关键理解**：`mybatis-plus-boot-starter` 让每个 Mapper 接口继承 `BaseMapper<T>` 就自动拥有 CRUD 能力，无需写一行 XML。

### 🔹 第 2 个：`ClubApplication.java`

**路径**：`com.club.ClubApplication`
**行数**：~20 行
**作用**：SpringBoot 启动入口

**阅读要点**：

- `@MapperScan` 告诉 MyBatis-Plus 去哪里找 Mapper 接口

## Phase 2：通用工具层（3 个文件）

> 📌 **目的**：理解整个后端的数据格式、认证机制、错误处理是怎么统一的

### 🔹 第 4 个：`Result.java` ⭐ 最基础

**路径**：`com.club.common.Result`
**行数**：~35 行
**依赖**：无（最底层，不被任何文件依赖）

**作用**：统一响应格式。后端所有接口返回的数据都包装成 `{code, msg, data}` 结构。

```java
Result<T> {
    int code;     // 200=成功, 400=客户端错误, 500=服务端错误
    String msg;   // 提示信息
    T data;       // 实际数据
}
```

**5 个静态工厂方法**：
- `Result.ok(data)` — 成功 + 数据
- `Result.ok()` — 成功无数据
- `Result.fail("msg")` — 失败 (code=400)
- `Result.fail(code, "msg")` — 自定义错误码
- `Result.error("msg")` — 服务器错误 (code=500)

**关键理解**：前端 `request.js` 的响应拦截器检查 `res.data.code === 200`，就是从这里来的。这是前后端契约。

---

### 🔹 第 5 个：`JwtUtil.java` ⭐ 认证核心

**路径**：`com.club.common.JwtUtil`
**行数**：~60 行
**依赖**：jjwt 库 + `application.yml` 中的 `jwt.secret` 和 `jwt.expiration`

**作用**：JWT 令牌的"签发-解析-验证"三件套

**3 个核心方法**：

| 方法 | 输入 | 输出 | 何时调用 |
|------|------|------|----------|
| `generateToken(userId, username, role)` | 用户信息 | JWT 字符串 | 登录成功时 |
| `getUserId(token)` | JWT 字符串 | 用户ID | 鉴权时 |
| `validate(token)` | JWT 字符串 | true/false | 请求到达时 |

**关键理解**：
- Token 的 payload 里存了 `userId` 和 `role`
- `SecurityConfig` 中的 `JwtAuthFilter` 每个请求都会调用 `validate()` 和 `getUserId()`
- 后端是无状态的 — 不存 session，完全靠 JWT 识别用户

---

### 🔹 第 6 个：`GlobalExceptionHandler.java`

**路径**：`com.club.common.GlobalExceptionHandler`
**行数**：~25 行
**依赖**：`Result.java`

**作用**：兜底异常处理。当 Controller 或 Service 抛出未捕获的异常时，自动转成 `Result` 格式返回，而不是暴露 Tomcat 的 500 错误页。

```java
@RestControllerAdvice  // 拦截所有 @RestController 的异常
public class GlobalExceptionHandler {
    @ExceptionHandler(Exception.class)       // 兜底
    @ExceptionHandler(RuntimeException.class) // 业务异常
}
```

**关键理解**：这个文件保证前端永远不会收到非 JSON 格式的错误响应。

## Phase 3：安全配置（1 个文件）

> 📌 **目的**：理解整个系统的权限体系是怎么运作的

### 🔹 第 7 个：`SecurityConfig.java` ⭐⭐⭐ 最重要

**路径**：`com.club.config.SecurityConfig`
**行数**：~95 行
**依赖**：`JwtUtil.java`

**作用**：Spring Security + JWT 整合配置。定义了什么接口需要什么权限、JWT 过滤器怎么工作、CORS 怎么配置。

**4 个核心部分**：

```java
// ① 放行规则（谁可以访问）
.antMatchers("/api/user/login", "/api/user/register").permitAll()   // 登录注册无需token
.antMatchers("/api/admin/**").hasAuthority("ADMIN")                 // 管理员专属
.anyRequest().authenticated()                                       // 其余都要登录

// ② 无状态会话（不用Cookie，每次请求都带Token）
.sessionManagement().sessionCreationPolicy(SessionCreationPolicy.STATELESS)

// ③ JWT 过滤器（在 Spring Security 之前执行）
.addFilterBefore(new JwtAuthFilter(jwtUtil), UsernamePasswordAuthenticationFilter.class)

// ④ CORS 配置（只允许前端 5173 跨域）
config.setAllowedOrigins(List.of("http://localhost:5173"));
```

**JwtAuthFilter 内部类的工作流程**（这是重点中的重点）：
```
每个 HTTP 请求到达
  → JwtAuthFilter.doFilter()
    → 从 Header 取 "Authorization: Bearer xxx"
      → JwtUtil.validate(token) 验证签名+过期
        → JwtUtil.getUserId(token) 提取用户ID
          → JwtUtil.getRole(token) 提取角色
            → 构造 Authentication 对象，放入 SecurityContext
              → 后续代码通过 SecurityContextHolder.getContext() 就能拿到当前用户
```

**关键理解**：
- 后续 Controller 中 `getCurrentUserId()` 和 `getCurrentUserRole()` 就是从 SecurityContext 中提取的
- 这是整个权限体系的基石 — 理解了这里，后面所有 `getCurrentUserId()` 就都明白了

## Phase 5：数据访问层（14 个 Mapper 文件）

### 21-34. 全部 Mapper 文件

**模式**：每个 Mapper 都是这样：
```java
@Mapper

```

| 序号 | Mapper 文件 | 对应 Entity |

**关键理解**：MyBatis-Plus 的 `BaseMapper<T>` 自动提供：
- `selectById(id)` — 按主键查
- `selectList(wrapper)` — 条件查询

## Phase 6：业务逻辑层（1 个文件）

> 📌 **目的**：理解核心业务逻辑 — 登录注册怎么实现的

### 🔹 第 36 个：`UserService.java` ⭐

**路径**：`com.club.service.UserService`
**行数**：~70 行
**依赖**：`UserMapper`、`JwtUtil`、`BCryptPasswordEncoder`

**3 个方法**：

| 方法 | 业务流程 |
|------|----------|
| `register(User)` | 校验空值 → 查重 → BCrypt 加密密码 → 默认角色 STUDENT → 入库 |
| `login(username, password)` | 查用户 → 校验状态 → BCrypt 验密 → JwtUtil 签发 Token → 返回用户信息+Token |
| `listUsers(keyword, page, size)` | 关键字模糊搜索 → 分页查询 → 返回 |

**关键理解**：
- 密码用 `BCryptPasswordEncoder` 加密，存进数据库的是 hash 值（如 `$2a$10$...`）
- 登录时用 `passwordEncoder.matches(明文, hash)` 验证，从不明文比较
- `lambdaQuery()` 是 MyBatis-Plus 的快捷查询方法，等效于 `this.list(new LambdaQueryWrapper<>())`

**贯穿所有接口的安全模式**（必看）：
```java
// 每个需要鉴权的接口都这样获取当前用户：
private Long getCurrentUserId() {
    Object principal = SecurityContextHolder.getContext().getAuthentication().getPrincipal();
    return principal instanceof Long ? (Long) principal : null;
}

private String getCurrentUserRole() {
    var auth = SecurityContextHolder.getContext().getAuthentication();
    // 从 authorities 中取第一个（即 role）
    return auth.getAuthorities().iterator().next().getAuthority();
}
```

## Phase 8：AI/LLM 模块（4 个文件）⭐ 创新亮点

> 📌 **目的**：理解 AI 智能策划是怎么实现的 — 答辩加分项
> 🔗 **调用链**：AiPlanController → AgentOrchestrator → ClaudeClient → DeepSeek API

### 🔹 第 40 个：`ClaudeClient.java` ⭐

**路径**：`com.club.llm.ClaudeClient`
**行数**：~175 行
**依赖**：OkHttp + `application.yml` 中的 `llm.*` 配置

**核心方法**：`sendMessage(systemPrompt, userMessage)`

**工作流程**：
```
1. 检查 API Key 是否配置
2. 根据 provider 构造不同的 HTTP 请求
   ├── deepseek: POST https://api.deepseek.com/v1/chat/completions
   │   格式: {model, messages: [{role:"system", content}, {role:"user", content}]}
   └── anthropic: POST https://api.anthropic.com/v1/messages
       格式: {model, system, messages: [{role:"user", content}]}
3. 发送请求（最多3次指数退避重试：等2s → 等4s → 等8s）
4. 解析响应
   ├── deepseek: choices[0].message.content
   └── anthropic: content[0].text
5. 返回文本内容
```

**双 Provider 架构**（答辩亮点）：
- 改 `application.yml` 中的 `llm.provider` 就能切换
- `buildDeepSeekRequest()` vs `buildAnthropicRequest()` — 两种 API 格式
- `parseOpenAIResponse()` vs `parseAnthropicResponse()` — 两种响应解析

---

### 🔹 第 41 个：`AgentOrchestrator.java` ⭐

**路径**：`com.club.llm.AgentOrchestrator`
**行数**：~85 行
**依赖**：`ClaudeClient`、`LlmCallLogMapper`、`LangfuseClient`

**核心方法**：`orchestrate(userRequirement, userId, activityId)`

**工作流程**：
```
1. 用精心设计的 PRD_PROMPT 作为 System Prompt
   → "你是一位经验丰富的校园活动策划专家..."
   → 约束 AI 按七个固定章节输出 Markdown

2. callAgentWithLog("PRD", systemPrompt, userRequirement, userId, activityId)
   ├── 记录开始时间
   ├── 创建 LlmCallLog 实体（userId, agentType, inputPrompt）
   ├── 调用 claudeClient.sendMessage()
   ├── 成功后：记录 outputContent, latencyMs, status=SUCCESS
   ├── 失败后：记录 errorMsg, status=FAILED
   └── 写入 llm_call_log 数据库表

3. MermaidUtil.extractMarkdown(raw) 清洗输出
4. 返回 {prd: "Markdown策划方案", totalTime: 耗时ms}
```

**PRD_PROMPT 的设计思路**（答辩重点）：
- 定义 AI 角色（"校园活动策划专家"）
- 约束输出格式（7个固定章节，Markdown 格式）
- 要求具体可操作（"直接给方案不要解释"）

### 🔹 第 43 个：`LangfuseClient.java`

**路径**：`com.club.llm.LangfuseClient`
**行数**：~115 行
**依赖**：无外部 SDK（纯 JDK HTTP）

**作用**：LLM 调用可观测性 — 把每次 AI 调用的耗时、token、成败上报到 Langfuse 平台

**核心方法**：`trace(traceName, userId, input, output, tokens, latencyMs, status)`

**工作流程**：
```
1. 检查 langfuse.enabled（默认 false，不上报）
2. 构造 JSON trace 数据
3. HTTP POST → Langfuse API（Basic Auth）
4. 3秒超时，失败不影响主流程
```

**关键理解**：
- 默认关闭（`enabled: false`），不配置也不影响系统运行
- 设计原则：埋点失败不能影响业务 — `catch (Exception e)` 只打日志
- 答辩时可以提"LLMOps 可观测性" — 有全链路追踪日志
